# 提示词模板集

本文档包含6个核心提示词模板，可直接复制给AI使用，快速生成高质量需求文档。

---

## 模板1：项目启动提示词

**使用场景**：项目初期，需要生成完整的需求文档初稿

**使用方法**：复制下面的模板，填入你的项目信息，发送给AI

```markdown
我要设计一个[项目类型]的需求文档，请帮我按照以下结构完成：

## 背景信息
- 项目名称：[填写项目名称]
- 核心问题：[要解决的痛点，例如：用户找PPT模板效率低]
- 目标用户：[用户画像，例如：职场人士、教师、学生]
- 竞品参考：[类似产品，例如：第一PPT、包图网]
- 技术基础：基于mksaas-template（Next.js + PostgreSQL + Better Auth + Stripe + Resend）

## 设计要求
1. 按照EARS格式编写验收标准（使用WHEN/THE/SHALL/IF/WHILE等关键词）
2. 功能按P0/P1/P2优先级分类
   - P0：MVP必需功能
   - P1：增强功能
   - P2：优化功能
3. 分4个阶段实施
   - 阶段0：MVP验证（1-2周）
   - 阶段1：智能化增强（2-3周）
   - 阶段2：商业化闭环（2周）
   - 阶段3：规模化优化（持续）
4. 每个阶段包含：时间估算、功能清单、验收标准、技术栈
5. 覆盖7个设计层面：
   - 前端层（UI/UX、组件库）
   - 后端层（API、数据库）
   - 交互层（状态管理、API契约）
   - 中间件层（异步任务、缓存）
   - SEO/AIO层（搜索优化、AI索引）
   - 部署层（环境变量、CI/CD）
   - 安全层（认证授权、数据保护）

## 输出格式
请按照以下章节输出完整文档：

### 1. 项目概述
- 1.1 业务背景（问题陈述、解决方案）
- 1.2 目标用户（用户画像、使用场景）
- 1.3 核心价值主张（差异化优势）
- 1.4 成功指标（DAU、转化率、收入等）

### 2. 功能需求
- 2.1 核心功能列表（按P0/P1/P2分类）
- 2.2 每个功能包含：
  - 用户故事（作为[角色]，我希望[功能]，以便[价值]）
  - 验收标准（EARS格式，5-8条）
  - 技术实现（前端、后端、依赖、风险）
  - 验证方式（测试、监控）

### 3. 技术架构
- 3.1 技术选型（复用mksaas-template的能力 + 新增技术栈）
- 3.2 系统架构图（用Mermaid或文字描述）
- 3.3 数据模型设计（Drizzle Schema格式）
- 3.4 API设计（端点、Request/Response）

### 4. 非功能需求
- 4.1 性能要求（响应时间、并发数）
- 4.2 安全要求（认证、授权、加密）
- 4.3 可观测性（日志、监控、告警）
- 4.4 SEO/AIO优化（Meta标签、Sitemap）

### 5. 实施计划
- 5.1 阶段0：MVP（时间、目标、功能清单、验收标准、技术栈）
- 5.2 阶段1：增强（同上）
- 5.3 阶段2：商业化（同上）
- 5.4 阶段3：规模化（同上）

### 6. 风险管理
- 6.1 技术风险（风险、影响、概率、应对策略）
- 6.2 业务风险（同上）
- 6.3 应对策略总结

### 7. 成本预估
- 7.1 开发成本（人力、第三方服务）
- 7.2 运营成本（月度，列出所有服务费用）
- 7.3 ROI分析（收入预测、回本周期）

### 8. 附录
- 8.1 参考资料
- 8.2 竞品分析
- 8.3 技术调研

请开始设计，输出完整的需求文档。
```

**示例填写**：
```markdown
- 项目名称：AI智能PPT模板推荐平台
- 核心问题：用户需要花费大量时间浏览和筛选PPT模板，传统关键词搜索无法理解用户真实需求
- 目标用户：职场人士（工作汇报）、教师（教学课件）、学生（学术答辩）
- 竞品参考：第一PPT（1ppt.com）、包图网（ibaotu.com）
```

## 模板2：功能需求细化提示词

**使用场景**：对某个具体功能进行详细设计

**使用方法**：为每个核心功能单独使用此模板

```markdown
我需要为[功能名称]编写详细的需求说明，请按照以下格式输出：

## 功能信息
- 功能名称：[例如：智能语义搜索]
- 所属模块：[例如：搜索模块]
- 优先级：P0/P1/P2（[说明理由]）

## 功能背景
- 用户故事：作为[角色]，我希望[功能]，以便[价值]
- 业务价值：[为什么做这个功能，解决什么问题]
- 使用场景：[列出2-3个具体场景]

## 验收标准（EARS格式）
请用以下句式编写5-8条验收标准：

1. **事件驱动型**：WHEN [触发条件]，THE [系统] SHALL [响应行为]
   示例：WHEN User输入查询词，THE Platform SHALL在3秒内返回结果

2. **普遍性型**：THE [系统] SHALL [必须满足的条件]
   示例：THE Platform SHALL支持中文和英文查询

3. **状态驱动型**：WHILE [状态条件]，THE [系统] SHALL [持续行为]
   示例：WHILE User未登录，THE Platform SHALL显示登录提示

4. **异常处理型**：IF [异常条件]，THEN THE [系统] SHALL [异常处理]
   示例：IF 搜索无结果，THEN THE Platform SHALL显示推荐模板

请根据功能特点编写5-8条验收标准。

## 技术实现
- **前端实现**：
  - 页面/组件：[例如：src/app/[locale]/search/page.tsx]
  - UI组件：[例如：SearchBar、ResultCard]
  - 状态管理：[例如：使用TanStack Query管理搜索状态]

- **后端实现**：
  - API端点：[例如：POST /api/search]
  - 数据库表：[例如：templates表]
  - 第三方服务：[例如：OpenAI Embedding API、Pinecone]

- **依赖关系**：
  - 前置依赖：[需要先完成的功能]
  - 后续依赖：[这个功能完成后才能做的功能]

- **技术风险**：
  - [可能的技术难点和应对方案]

## 数据模型（如涉及）
```typescript
// 用Drizzle Schema格式定义相关表结构
export const tableName = pgTable('table_name', {
  // 字段定义
});
```

## API设计（如涉及）
```typescript
// Request
POST /api/endpoint
{
  "field1": "value1",
  "field2": "value2"
}

// Response
{
  "data": [...],
  "total": 100,
  "error": null
}
```

## 验证方式
- **单元测试**：[测试什么逻辑]
- **集成测试**：[测试什么流程]
- **用户验证**：[如何让用户验证，例如：邀请10个用户测试]
- **监控指标**：[如何观测效果，例如：PostHog追踪搜索词和点击率]

## 成功指标
- [量化指标1，例如：搜索准确率 > 80%]
- [量化指标2，例如：平均响应时间 < 3秒]
- [量化指标3，例如：用户满意度 > 4.0/5.0]

请开始设计，输出完整的功能需求文档。
```

**示例填写**：
```markdown
- 功能名称：智能语义搜索
- 所属模块：搜索模块
- 优先级：P0（核心功能，没有搜索用户无法使用）
```

## 模板3：技术架构设计提示词

**使用场景**：确定技术选型和系统架构

**使用方法**：在功能需求明确后使用

```markdown
我需要为[项目名称]设计技术架构，基础是mksaas-template，请帮我完成：

## 现有能力分析
mksaas-template已提供以下能力：
- **前端**：Next.js 15 + React 19 + shadcn/ui + Tailwind CSS

- **数据库**：PostgreSQL + Drizzle ORM
- **支付**：Stripe
- **邮件**：Resend + React Email
- **国际化**：next-intl
- **分析**：Vercel Analytics + PostHog + OpenPanel
- **AI**：Vercel AI SDK（支持OpenAI、DeepSeek等）
- **部署**：Vercel（main分支）或Cloudflare Workers（cloudflare分支）

## 新增需求
我需要实现以下功能：
1. [功能1，例如：语义搜索]
2. [功能2，例如：文件存储]
3. [功能3，例如：异步任务处理]

## 设计要求
请输出以下内容：

### 1. 技术选型对比表
对每个新增需求，列出3个候选方案，对比：
- 优点
- 缺点
- 成本（免费层 + 付费层）
- 学习曲线
- 与mksaas-template的集成难度
- 推荐方案及理由

示例格式：
| 需求 | 方案1 | 方案2 | 方案3 | 推荐 |
|------|-------|-------|-------|------|
| 向量搜索 | Pinecone | Qdrant | Milvus | Pinecone（免费层足够MVP） |

### 2. 系统架构图
用Mermaid或ASCII画出：
- 前端层（页面、组件）
- API层（Next.js API Routes/Server Actions）
- 数据层（PostgreSQL、向量数据库）
- 存储层（对象存储）
- 第三方服务（AI API、支付、邮件）
- 异步任务层（队列、定时任务）

### 3. 数据模型设计
用Drizzle Schema格式定义所有表结构：
```typescript

  id: uuid('id').primaryKey().defaultRandom(),
  // 其他字段
  createdAt: timestamp('created_at').defaultNow(),
  updatedAt: timestamp('updated_at').defaultNow(),
});
```

包含：
- 表名和字段定义
- 主键和外键关系
- 索引设计
- 与Better Auth的集成（如需要）

### 4. API设计
列出所有API端点：
```typescript
// 端点：POST /api/[endpoint]
// 功能：[描述]
// 认证：[是否需要登录]

Request: {
  // 请求参数
}

Response: {
  // 响应格式
}

// 错误处理
Error: {
  "error": "错误信息",
  "code": "ERROR_CODE"
}
```

### 5. 部署方案
说明：
- 如何在Vercel上部署（环境变量、数据库连接）
- 如何在Cloudflare上部署（如需要）
- 数据库迁移策略（Drizzle Kit）
- 环境管理（开发、预览、生产）

### 6. 成本估算
列出所有第三方服务的费用：

| 服务 | 免费层 | 付费层 | MVP阶段 | 规模化阶段 |
|------|--------|--------|---------|------------|
| Vercel | 免费 | $20/月 | 免费 | Pro |
| PostgreSQL | Supabase免费500MB | $25/月 | 免费 | Pro |
| ... | ... | ... | ... | ... |

总计：
- MVP阶段：约$X/月
- 规模化阶段：约$Y/月

请开始设计，输出完整的技术架构文档。
```

**使用方法**：在技术架构确定后使用

## 阶段划分原则

**技术栈**：
- 复用能力：[列出复用mksaas-template的哪些能力]

- [风险1] → [应对方案]

- [这个阶段完成后，下一阶段才能做什么]

### 阶段3：规模化优化（持续迭代）

请开始设计，输出完整的阶段规划文档。

## 模板5：风险评估提示词

**使用场景**：识别和应对项目风险

**使用方法**：在规划阶段使用

```markdown
我需要为[项目名称]做风险评估，请帮我识别和应对风险：

## 项目背景
- 技术栈：[列出主要技术，例如：Next.js + PostgreSQL + Pinecone]
- 团队规模：[人数和技能，例如：1人全栈，熟悉React和Node.js]
- 时间限制：[上线deadline，例如：4周后]
- 预算限制：[成本上限，例如：月运营成本<$100]
- 目标规模：[用户规模，例如：DAU 1000]

## 风险识别
请从以下维度识别风险：

### 1. 技术风险
- 技术选型风险（新技术学习成本、兼容性）
- 性能瓶颈风险（响应时间、并发能力）
- 第三方依赖风险（API限流、服务稳定性）
- 数据安全风险（数据泄露、攻击）
- 技术债务风险（代码质量、可维护性）

### 2. 业务风险
- 用户增长风险（获客成本、留存率）
- 变现能力风险（付费意愿、竞争压力）
- 内容版权风险（侵权、DMCA）
- 合规风险（隐私保护、数据安全）
- 竞品压力风险（功能对比、差异化）

### 3. 团队风险
- 人员流动风险（关键人员离职）
- 技能缺口风险（缺少某项技能）
- 沟通成本风险（远程协作、时区差异）
- 进度延期风险（估算不准、需求变更）

### 4. 成本风险
- 开发成本超支（时间延期、人力增加）
- 运营成本超支（用户增长、服务升级）
- 第三方服务涨价（API费用、托管费用）

## 输出格式
请用表格输出：

| 风险类型 | 具体风险 | 影响程度 | 发生概率 | 应对策略 | 负责人 | 预算 |
|---------|---------|---------|---------|---------|--------|------|
| 技术 | [风险描述] | 高/中/低 | 高/中/低 | [具体措施] | [角色] | [金额] |

并为每个**高影响**或**高概率**的风险制定详细的应对计划：

### 风险：[风险名称]
- **影响**：[详细说明影响范围和程度]
- **概率**：[为什么可能发生]
- **应对策略**：
  1. 预防措施：[如何降低发生概率]
  2. 应急预案：[如果发生了怎么办]
  3. 监控指标：[如何及时发现]
  4. 负责人：[谁来执行]
  5. 预算：[需要多少资源]

## 风险优先级矩阵
请用矩阵展示风险优先级：

```
高影响 | [风险A] | [风险B] |
      |---------|---------|
中影响 | [风险C] | [风险D] |
      |---------|---------|
低影响 | [风险E] | [风险F] |
      |---------|---------|
       低概率    中概率    高概率
```

优先处理：高影响+高概率的风险

请开始评估，输出完整的风险管理文档。
```
